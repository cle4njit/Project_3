Pytest-
Pytest is a method used in creating projects which can make tests to ensure the project works fine. The most important files utilizing Pytest in this project were the simple_pages_test.py files and the context_process_test.py file. The context test file was used to test the flask environment. Meanwhile, the simple_pages_test tested that every page returned a 200 status code. These simple pages in turn were created via first extending base.html and then having local content which was separated with block content and endblock.

Project Outline Explanation

DockerFile (root folder): Let's the user know what extra features need to be downloaded.


Readme.md (root folder): Used by Github to explain the repository/project that's been created by the user.


Requirements.txt (root folder): Used by the DockerFile to install specific features, such as pytest and Flask.


dev.yml (root folder | .github folder | workflows folder): The development workflow file, it details that when a pull request to the master branch comes through, python 3.8 is installed alongside the features in requirements.txt and runs pytest. After that, it deploys to the dev app of the project in heroku utilizing the api key and the author's username and email.


prod.yml (root folder | .github folder | workflows folder): The production workflow file, it details that when a push request to the master branch comes through, python 3.8 is installed alongside the features in requirements.txt and runs pytest. After that, it also builds a container on Docker via utilizing the author's username and password as well as extracts the user's image(s) in Docker. Finally, it deploys to the prod app of the project into heroku utilizing the api key and the author's username and email.


__init__.py (root folder | app folder | context_processors folder): Utilized by the base.html file, this file has a message created by the developer which is used in the copyright message on the website.


__init__.py (root folder | app folder | simple_pages folder): Used by the project, this file imports needed functions from flask and jinja2, allowing for easier implementation of different pages as well as returning a 404 status error if said page cannot be found.


page1.html, page2.html, page3.html, page4.html & index.html (root folder | app folder | simple_pages folder | templates folder): These are all used in the project, this directory and the files within all extend the base.html file and has a block content file within to provide specified content.


style.css (root folder | app folder | static folder | css folder): In use by the html files, this file provides a small aesthetic detail to each html file (larger, 200% sized text).


base.html (root folder | app folder | templates folder): Utilized by the html files, this file provides a base template for each page on the website, allowing for a uniform look and a usable navigation bar.


__init__.py (root folder | app folder): Importing flask, the simple_pages directory and the context processors, this file has a function of create_app which creates and configures an instance of the flask app with a secret key and registering a 'blueprint' from the simple_pages directory.


run.py (root folder | app folder): Taking create_app from app, this file is only needed for configuration/ working with Gunicorn in order to serve the app in production in heroku.


conftest.py (root folder | tests folder): Importing pytest and create_app from the app, this file defines the following functions: Application (making the app itself), client (makes the http client) and runner (makes the task runner).


context_process_test.py (root folder | tests folder): Importing datetime and getenv from the operating system, this file tests if the environment is actually working for Flask (200 status code response) and tests the two different data types used in the copyright message (data and currency).


simple_pages_test.py (root folder | tests folder): Testing the website, this file first ensures that the navigation bar is properly named and goes to the right link, as well as ensures that every page does exist for the website (200 status code). Finally, it checks if a page that should not exist does properly return a 404 status code.
